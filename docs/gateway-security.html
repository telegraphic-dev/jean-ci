Gateway security

Keep the OpenClaw gateway private and boring.

jean-ci only needs a stable internal path and a valid bearer token. Public exposure, oversized token sharing, and ad-hoc proxying are where avoidable mistakes happen.

Recommended connection shape

Recommended environment values
OPENCLAW_GATEWAY_URL=http://coolify-proxy/openclaw
OPENCLAW_GATEWAY_TOKEN=replace-me

Why this shape

It keeps the gateway behind a private proxy path instead of exposing it directly to the internet. jean-ci only needs HTTP access to the internal bridge plus a bearer token.

Least-privilege habits

  • Use a dedicated token for jean-ci.
  • Keep the token in env vars or secret storage, not in docs or prompt files.
  • Prefer an internal proxy path or private network route.
  • Rotate tokens when staff or infrastructure changes.

Don’t do this

  • Do not expose the gateway on a public interface.
  • Do not reuse a broad admin token across unrelated services.
  • Do not assume a 200 from the app means the token is valid for reviews.

Safe rotation workflow

  1. Create a new gateway token.
  2. Deploy jean-ci with the new token.
  3. Verify health and run one sample PR review.
  4. Revoke the old token only after the new flow works.
Verify gateway access
curl -s -H "Authorization: Bearer $OPENCLAW_GATEWAY_TOKEN" "$OPENCLAW_GATEWAY_URL/health"
The best gateway setup is the one nobody notices because it stays private, stable, and easy to rotate.
