Prompt handbook

Write custom checks that stay sharp instead of going vague.

Good jean-ci prompts are narrow, deterministic, and explicit about PASS versus FAIL. The built-in Code Review already covers general review; custom prompts should add a focused rule, not re-describe “review the whole PR”.

Schema Principles OpenClaw prompts Examples

Required schema for custom checks

Every file in .jean-ci/pr-checks/ should follow this shape.

Minimal valid jean-ci prompt
# Prompt Title

## Purpose
One short paragraph explaining what this check protects.

## Review Instructions
Concrete things to inspect in the PR diff.
Tell the reviewer what counts as a blocking issue.

## Verdict Criteria
- **FAIL** when ...
- **PASS** when ...

Why this shape exists

Structured prompts are easier to validate, easier to read in-repo, and easier to keep deterministic across many PRs.

Prompt principles that hold up in production

One concern per prompt

Prefer security, migration safety, or docs coverage. Do not create a custom prompt that just says “review everything”. The built-in review already does that job.

Fail only for merge blockers

If a prompt fails for style nits or optional cleanup, it will train your team to ignore it.

Reference changed code paths

Tell the reviewer to anchor findings to what changed. Generic advice without changed-file grounding is less useful in PR checks.

State PASS and FAIL explicitly

Ambiguous verdict criteria create noisy or unstable output. Say exactly what should fail the PR.

OpenClaw-specific copy-paste prompts

These are for asking an OpenClaw agent to generate prompt files in another repo. They are intentionally explicit about the built-in review remaining in place.

Ask OpenClaw to add a migration-safety check
Add a jean-ci custom PR review check to this repository.

Requirements:
- Do not change or disable the built-in default `jean-ci / Code Review`.
- Add a new additive file at `.jean-ci/pr-checks/migration-safety.md`.
- Use the standard jean-ci prompt structure with `## Purpose`, `## Review Instructions`, and `## Verdict Criteria`.
- Scope the prompt to schema, data migration, rollout sequencing, backfill safety, and rollback safety.
- Fail only for merge-blocking migration risks.
- Commit the change and include a short PR note that custom checks are additive to the default review.
Ask OpenClaw to add tests-and-docs coverage
Add an additive jean-ci PR check at `.jean-ci/pr-checks/tests-and-docs.md`.

Rules:
- Keep the built-in `jean-ci / Code Review` as the always-on default review.
- The new custom prompt should check for missing tests or missing user-facing docs only when they are genuinely required by the diff.
- Do not fail for tiny internal refactors that do not need tests or docs updates.
- Use the jean-ci structured prompt format.
- Commit on a fresh branch and open a PR.

Example custom prompt: migration safety

Example migration-safety.md
# Migration Safety Review

## Purpose
Catch merge-blocking rollout and migration risks before a schema or data change lands.

## Review Instructions
Review the PR diff for incompatible schema changes, destructive migrations without a safe rollout path, missing backfill sequencing, irreversible data transformations, and rollback traps.
Reference the changed files when you find a blocking issue.
Only fail for real rollout or migration risks.

## Verdict Criteria
- **FAIL** when the diff introduces a schema or data migration change that can break production rollout, corrupt data, or block safe rollback.
- **PASS** when the migration path is operationally safe and no merge-blocking migration risks are present in the diff.
Custom prompts should be boring in the best way: narrow scope, clear fail conditions, easy to trust.
